## 🔴 Blockers (regressions this PR introduces — fix before merge)

### 1. [x] Experiment tracker deadlocks after an experiment ends — CONFIRMED
`backend/services/recommender.py:333` (`_get_active_experiment`)

Making reads idempotent (T-02) removed auto-complete-on-read, but `_get_active_experiment` still returns a past-due `ACTIVE` row as `active_experiment`, with effective status `"completed"`. The SPA then:
- disables **every** "Start experiment" button — `RecommendationsPage.tsx:45` (`hasActiveExperiment = data.active_experiment !== null`) → `:87` (`canStartExperiment={!hasActiveExperiment && …}`), and
- hides Complete/Abandon — `ExperimentTracker.tsx:92` renders them only when `experiment.status === "active"`, but the status is now `"completed"`.

**Failure scenario:** an experiment's `end_date` passes. On every `GET /api/recommendations`, `active_experiment` is non-null → all Start buttons disabled, and the tracker shows no action buttons → the user can neither clear the finished experiment nor start a new one. Pre-PR the first read auto-completed the row and the next read returned `active_experiment = null`, so the slot cleared. `complete_stale_experiments` is reachable only from `create_experiment`, which the UI can no longer trigger.

**Fix:** in `_get_active_experiment`, return `None` when `experiment.end_date < today` (a past-due experiment is not *currently* active). It still appears in `list_experiments` history and is persisted-complete on the next `create` (which runs `complete_stale_experiments` before the 409 check). Add a regression test: past-due active → `active_experiment` null and a new create succeeds.

### 2. [x] T-05 422 wrapper leaks SQL once T-09 makes a DB error reachable — CONFIRMED
`backend/routers/daily_log.py:183` (`update_entry`), same pattern at `:161` (`add_entry`)

`update_entry`/`add_entry` do `except Exception → HTTPException(422, detail=str(exc))`. `RedLightEntryCreate.panel_id` is `int | None` with no validation (`schemas.py:257`), and T-09 (this PR) now enforces the `red_light_entries.panel_id` FK, so `PUT …/red-light/{id}` (or `POST …/red-light`) with a nonexistent `panel_id` raises `sqlalchemy.IntegrityError` at `db.commit()`. The broad catch returns it as `422 detail="(sqlite3.IntegrityError) FOREIGN KEY constraint failed [SQL: UPDATE red_light_entries SET panel_id=? …] [parameters: …]"` — leaking schema/SQL internals and misclassifying a 500-class fault as a client 422. T-09 turned a latent leak (pre-existing broad catch on the add path) into a reachable one.

**Fix:** catch only `pydantic.ValidationError` for the 422; let integrity/DB errors surface as a clean 409/400 (or a generic 500) without echoing `str(exc)`.

### 3. [x] `_harden_db_permissions` chmods an arbitrary parent directory to 0700 — CONFIRMED
`backend/database.py:74`

`init_db` → `_harden_db_permissions(settings.db_path)` runs `os.chmod(db_path.parent, 0o700)` on every startup. For a custom `SOMNUS_DB_PATH` — the flow **T-07 explicitly recommends** (put the DB on an encrypted volume) — the parent may be a shared/general-purpose directory, silently locked to owner-only; a relative `SOMNUS_DB_PATH=somnus.db` resolves the parent to `.` and chmods the current working directory.

**Failure scenario:** `SOMNUS_DB_PATH=/home/user/shared-project/somnus.db` (a `0775` group-shared dir) becomes `0700` at launch, cutting off collaborators/other processes from the whole directory.

**Fix:** chmod only the DB *file* to `0600`; restrict `0700` dir-hardening to a directory we created (the default `~/.somnus`), e.g. only harden the parent when `db_path` is the default location.

## 🟠 Correctness of the mitigations / false claims (should fix before merge)

### 4. [x] `frame-ancestors 'none'` is a no-op in a `<meta>`-delivered CSP — CONFIRMED
`frontend/vite.config.ts:26`

Per the CSP spec, `frame-ancestors` (like `sandbox`/`report-uri`) is **ignored** when the policy is delivered via a `<meta>` element; browsers log `The Content Security Policy directive 'frame-ancestors' is ignored when delivered via a <meta> element` and enforce nothing. `docs/THREAT_MODEL.md` T-14 implies framing protection that does not exist — the exact "doc misstates code" failure the gate exists to prevent. (Found independently by two angles.)

**Fix:** remove `frame-ancestors` from the meta CSP; correct the T-14 text to note framing protection needs a response-header layer (from whatever serves `dist/`) or record it as an accepted residual alongside the T-01 packaged-build residual.

### 5. [x] CSP `<meta>` injected at the end of `<head>` — CONFIRMED
`frontend/vite.config.ts:36`

`injectTo: 'head'` appends the meta **after** the entry `<script type=module>` and stylesheet `<link>` in the built `dist/index.html`. A meta CSP only governs fetches initiated after it is parsed, so the entry script/stylesheet/favicon are not covered.

**Fix:** `injectTo: 'head-prepend'`.

### 6. [x] Global default `Content-Type: application/json` on the TestClient masks real paths — CONFIRMED
`backend/tests/conftest.py:47`

Setting the header client-wide makes **every** test request send JSON, including GETs and (empirically on this env's httpx) overriding multipart/`files=` bodies. Consequences: the SPA's real no-content-type navigation paths (report/export `<a href>` downloads) are never exercised, so if `require_json_content_type` were ever misapplied to a navigation-fetched endpoint the suite would still pass while users get 415; and the PLAN "data import" critical path's future `UploadFile` endpoint would receive JSON forced over its multipart body — upload tests fail confusingly, and negative-path tests pass for the wrong reason (JSON-parse 422 rather than the intended validation).

**Fix:** drop the global header; set `Content-Type: application/json` per-request only in the CSRF tests that need it (the SPA-representative default can stay if narrowed to the endpoints under test).

### 7. [x] `npm audit` runs after `npm ci` — CONFIRMED (convention)
`.github/workflows/ci.yml:87`

The gate runs after install, so dependency lifecycle (`postinstall`) scripts execute before the audit — inverting CLAUDE.md's **audit-before-install** rule. `npm audit --package-lock-only --omit=dev --audit-level=high` evaluates the lockfile with no `node_modules` (verified), so a check can run **before** `npm ci`.

**Fix:** add a `--package-lock-only` audit step before `npm ci` (keep or drop the post-install one).

### 8. [x] `ARCHITECTURE.md` not updated — CONFIRMED (CLAUDE.md violation)
`ARCHITECTURE.md` (data-flow line for Oura sync; backend component diagram)

CLAUDE.md: "ARCHITECTURE.md must be updated in the same commit as any structural change." This PR converts Oura sync `GET`→`POST` and adds a new cross-cutting module (`backend/security.py`, now a dependency of two routers), but ARCHITECTURE.md still documents `GET /api/oura/sync` and has no security component/edges. The PR's "Threat model impact" section is present and `THREAT_MODEL.md` is updated in-PR; this is the one doc-currency rule the PR misses.

**Fix:** update the sync data-flow to POST and add the `security` module (+ its two router edges) to the backend component diagram.

## 🟡 Lower-severity notes (reasonable follow-ups)

- **Altitude — CSRF guard is per-endpoint** (`backend/security.py`): enforced by author memory, not structure. The current surface is fully covered (verified), but a *future* bodiless mutating POST could ship without the dependency and silently reopen T-02 while `THREAT_MODEL` still says "Mitigated." A POST-scoped structural guard (middleware) would be SPA-transparent (both fetch helpers always send `application/json`) and subsume FastAPI's non-JSON-422 for body-taking POSTs. Trade-off: needs the TestClient default header (finding 6) handled coherently and the `SOMNUS_TESTING` reset endpoint exempted.
- **FK pragma binds the global `Engine`** (`backend/database.py:18`): Alembic's migration engine inherits it, so a future SQLite `batch_alter_table` (column alter → table recreate) will fail unless it sets `foreign_keys=OFF`. No current breakage (001/002 are additive). Add a note in `alembic/env.py` or scope the listener.
- **Settings defaults duplicated** (`backend/routers/settings.py:72`): `_settings_for_read` hand-copies five model column defaults, and `test_get_settings_defaults` asserts the same literals — so the test checks the copy against itself, not the model. A model-default change would pass CI while the fresh-install GET diverges from the row the first PATCH creates. Consider deriving defaults from `UserSettings.__table__.columns`, or add a drift-guard test that compares against a freshly-created persisted row.
- **Duplication** (`backend/routers/oura.py:27`): `_get_settings` is a byte-twin of `settings.py:_get_or_create_settings`; this PR made them semantically load-bearing (read/write split) but re-documented only one. Consolidate into a shared write-path helper.
- **Built app is never exercised** (`frontend/vite.config.ts:30`): the CSP is `apply: 'build'`, but unit tests (jsdom) and Playwright (dev server) never run the built bundle, so a future `blob:` export / web worker / external asset would break only in the user's production build. Consider one CSP smoke test against `vite preview`.
