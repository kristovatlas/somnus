## Verified sound (no security finding)

- **T-02 CSRF guard** (`backend/security.py`) — Exact-match on `application/json` (a CORS non-simple content-type); absent / `text/plain` / form content-types all → 415, so no cross-origin *simple* request can reach the two guarded bodiless POSTs (`POST /api/oura/sync`, `POST …/copy-from/…`). The defense rests on CORS being pinned to `http://localhost:5173` with `allow_credentials=True` and **no wildcard** — verified in `backend/main.py:34-40` / `backend/config.py:37`, and **unchanged by this PR** (`git diff origin/dev...HEAD -- backend/main.py backend/config.py` is empty). No newly-added/changed bodiless (CORS-simple-able) state-changer is left unguarded. No bypass.
- **T-12 CSV neutralization** (`backend/routers/export.py`) — `_neutralize_csv_cell` is applied to every data cell in `_to_csv`, the only CSV writer; the header row emits hardcoded column names, not user input. No attacker-controllable cell reaches a CSV un-neutralized. Leading-character detection is the correct semantics for formula injection.
- **T-17 SQLite snapshot** (`backend/routers/export.py`) — `db_path` is trusted config (`app_settings.db_path`), never request-influenced, so no path traversal; the online backup API (`.backup()` / `.serialize()`) executes no user SQL; `Content-Disposition` filenames are literals. The whole-DB `GET /export/sqlite` pre-dates this PR (unchanged behavior), so its data-exposure profile is not introduced here.
- **T-02 settings/recommender** — `_settings_for_read` returns hardcoded defaults on GET; the output schema (`UserSettingsOut`) exposes only `oura_token_set: bool`, never the raw Oura token, so no token can leak through this path. The recommender change moves a status write *off* GET paths — a net reduction in CSRF/idempotency surface, no injection or exposure.
- **T-08/T-09 DB hardening** (`backend/database.py`) — `os.chmod` only *tightens* permissions (0700/0600); `PRAGMA foreign_keys=ON` is an integrity control. No user input, no confidentiality/integrity exposure. (The companion review's blocker #3 on the parent-dir chmod is a real usability/availability footgun on custom `SOMNUS_DB_PATH`, but tightening permissions cannot itself be a confidentiality/integrity vulnerability — so it is out of scope for this security pass and correctly owned by the companion review.)
- **T-14 CSP** (`frontend/vite.config.ts`) — Defense-in-depth; `script-src 'self'` with no `unsafe-inline`/`eval` (the `unsafe-inline` is confined to `style-src`, low-impact). CSP weaknesses are hardening/defense-in-depth, out of scope here; the companion review's findings 4/5 (meta `frame-ancestors` no-op, injection ordering) correctly track these as mitigation-fidelity/doc issues, not exploitable holes.
- **T-13 supply-chain** (`.github/workflows/ci.yml`, `frontend/package*.json`) — SHA-pinned Actions and CI `npm audit` are improvements; the react-router bump is a dependency-version change (out of scope for this review).

## Overlap with the companion review — T-05 broad `except` (companion blocker #2)

The companion review's blocker #2 flags `except Exception → HTTPException(422, detail=str(exc))` in `backend/routers/daily_log.py` (update path `:183`, mirrored on the add path `:161`): with T-09 now enforcing the `red_light_entries.panel_id` FK, a bad `panel_id` raises `sqlalchemy.IntegrityError` at commit, and the broad catch echoes the SQL statement + bound parameters into the 422 body.

**Security assessment (this pass): agree it should be fixed, but it is NOT an exploitable data-exposure vulnerability.** No trust boundary is crossed:
- The app is single-user / single-tenant and runs on localhost; the only party entitled to read the response already owns all the data and the (open-source) schema. The leaked text is a SQL template plus the caller's *own* submitted parameters — not another tenant's data, not a secret. `update_sub_entry` never touches the settings/token table.
- A cross-origin attacker cannot read the response body: the `PUT …/red-light/{id}` variant is a non-simple method (preflight-blocked by the pinned CORS), and the `POST …/red-light` variant, even if reached cross-origin via the no-`Content-Type` trick, returns an **opaque** response the attacker cannot read.

So this does not meet the `/security-review` exploitability bar and is not raised here as a security finding. I concur with the companion review's remediation on **robustness/correctness** grounds: catch only `pydantic.ValidationError` for the 422 and let integrity/DB faults surface as a clean 409/400 (or generic 500) without echoing `str(exc)`. That fix also removes any future scenario (e.g. a hypothetical multi-user deployment) where the echo *would* matter — cheap defense-in-depth.

## Informational — pre-existing, NOT introduced by this PR (non-blocking)

FastAPI parses a request body as JSON when the `Content-Type` header is **absent** (`fastapi/routing.py`: `if not content_type_value: json_body = await request.json()`). So body-bearing POSTs that pre-date this PR (`POST /api/experiments`, sub-entry add, red-light-panel create) remain cross-site reachable via a no-`Content-Type` simple request — e.g. `fetch(url, {method:"POST", body:new Blob([json],{type:""})})` — with no CORS preflight. The mechanism is real, but these endpoints' decorators/signatures are unchanged context in this PR, and the two endpoints PR #65 hardened are immune (the guard 415s the absent-`Content-Type` case). This is an existing concern, not a regression in #65. Optional defense-in-depth for a future PR (also raised as an "altitude" note in the companion review): apply `require_json_content_type` to the JSON-body POSTs too, or move the content-type check into a POST-scoped middleware.
