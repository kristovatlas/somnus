# ADR 014: Dependency-Install Cooldown (Minimum Release Age)

## Status
Accepted ŌĆö 2026-07-06. Implements part of threat-model **T-13** (PLAN.md Step 9.3).

## Context
Somnus holds maximally sensitive data behind an unauthenticated local API; a compromised dependency (AD3 in `docs/THREAT_MODEL.md`) runs with build/run privileges. The dominant real-world supply-chain attack is **publish-then-yank**: an attacker publishes a malicious version (account takeover / typosquat / maintainer compromise) that is usually detected and removed within hours to a few days. Consuming releases immediately ŌĆö as `pip install` (floating `>=`) and `npm install` do ŌĆö maximizes exposure to that window. T-13 flagged the absence of any minimum-release-age ("cooldown") gate as an Open gap.

By 2026 the major package managers ship native release-age gates: npm `min-release-age` (11.10+), pnpm/yarn/bun equivalents, and uv `--exclude-newer` with relative durations (0.9.17+). Somnus uses npm (frontend) and pip (backend).

## Decision
Enforce a **~7-day minimum release age** on dependency installs in both ecosystems.

1. **Frontend (npm):** `min-release-age=7` in `frontend/.npmrc` (value is in **days** ŌĆö npm semantics; pnpm/bun equivalents use minutes/seconds). It gates `npm install` / `npm update` resolution; frozen `npm ci` (the CI install) is unaffected ŌĆö correct, since the vetting happens when the lockfile is updated. Because npm < 11.10 **silently ignores** the key, the floor is enforced where the gate actually acts (local lockfile updates): `"engines": {"npm": ">=11.10.0"}` in `frontend/package.json` plus `engine-strict=true` in `.npmrc` make an install on old npm a hard error instead of a silent bypass. CI upgrades npm so `npm ci` passes the same engines check.
2. **Backend (uv):** the cooldown is **committed in `pyproject.toml`** as top-level `[tool.uv] exclude-newer = "7 days"` ŌĆö the single cooldown key, governing `uv lock` regeneration *and* the entire `uv pip` interface (both verified on the pinned uv 0.11.26; an earlier draft claimed uv config accepts only a fixed RFC-3339 timestamp, which is wrong ŌĆö relative durations apply). *(History: the key originally lived at `[tool.uv.pip]`; when `uv.lock` landed on 2026-07-15 it moved to `[tool.uv]` and the pip-level duplicate was removed, because on divergence the pip-section value silently wins for installs while `uv lock` follows the top level ŌĆö one policy, one key.)* Since the lockfile landed, the cooldown's job is done at **lock time**: `make setup-backend` reproduces the committed, hash-verified resolution and deliberately runs its locked install with the cooldown disabled ŌĆö the same division of labor as npm's, where `min-release-age` gates lockfile updates and frozen `npm ci` is unaffected. The install recipe lives once, in `make setup-backend`, which CI runs; outside a virtualenv the recipe passes `--system`, matching the old pip behavior (the README quickstart runs `make setup` with no venv active). uv replaces pip for backend installs ŌĆö it is pip-compatible and the only backend resolver with a rolling cooldown.
3. **The gating tools are pinned, not floating:** `uv==<UV_VERSION>` in the Makefile and `npm@11.18.0` (exact) in CI. An unpinned `pip install uv` / `npm install -g npm@^11.10` would fetch the newest release with no cooldown ŌĆö reintroducing the publish-then-yank vector through the very tools that enforce the control. Pins are bumped via reviewed PRs, which is a de facto cooldown for the tools themselves.
4. **Security-fix override:** for the frontend, an `npm install --before` bypass. For the locked backend, the working path is a **committed per-package override**: add `[tool.uv] exclude-newer-package = { <pkg> = "0 days" }` to `pyproject.toml`, run `uv lock`, and PR it (verified on uv 0.11.26: the override is recorded in `uv.lock` `[options]` and `uv lock --check` passes); remove it once the release ages past the window. *The `UV_EXCLUDE_NEWER` env var is NOT the emergency path anymore* ŌĆö `make setup-backend`'s locked install never re-resolves (the env var is a no-op there), and a lock regenerated under the env var records `exclude-newer-span = PT0S`, failing CI's `uv lock --check`. (An *empty* env value is a parse error; but you shouldn't be reaching for it at all.)
5. **Deferred (remaining T-13):** a committed backend lockfile (`uv.lock` via `uv sync`), `npm audit` in CI, and pinning GitHub Actions by SHA ŌĆö tracked in T-13, not in this change. *Status update (2026-07-09): `npm audit` in CI (lockfile-only, run before `npm ci`) and Action SHA-pinning landed in PR #65.* *Status update (2026-07-15): the backend lockfile landed ŌĆö `uv.lock` committed (including the setuptools build backend), installed via `uv export --locked` ŌåÆ hash-required `uv pip install --require-hashes -r` ŌåÆ `--no-deps --no-build-isolation` editable install, rather than the `uv sync` this ADR anticipated (uv sync only targets a project venv ŌĆö no `--system` mode ŌĆö so it can't serve the CI/README no-venv flow; `--inexact` addresses package-stripping, not env-targeting). `--locked` fails the local install loudly on a stale lock; CI's `uv lock --check` double-covers it. Cooldown consolidated to the single top-level `[tool.uv]` key (┬¦2) with the per-package emergency path (┬¦4). All T-13 sub-items are now done.*

## Consequences
**Positive:** closes the publish-then-yank window on both ecosystems; native gates are the primary control (Socket PR alerts remain as malware detection); uv also speeds up backend installs.

**Negative / costs:** contributors need uv (the Makefile bootstraps a pinned version) and npm Ōēź 11.10 (`engine-strict` fails fast with an upgrade hint; CI provisions it). Legitimate releases lag by ~7 days ŌĆö mitigated by the override path for security fixes. CI caches `~/.cache/uv` + `~/.cache/pip` (keyed on `pyproject.toml` + `uv.lock` + `Makefile`) so the uv switch doesn't lose the old pip cache's speed. ~~The backend remains unpinned (no lockfile yet): the cooldown bounds *how new* a version may be, not *which exact* one resolves ŌĆö the lockfile is the deferred complement.~~ *(Resolved 2026-07-15: `uv.lock` committed ŌĆö see status update above.)*
